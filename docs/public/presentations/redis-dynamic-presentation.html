Redis Engineering Briefing

Redis 不是缓存。它是高性能状态层。

从缓存、计数器、排行榜到分布式锁，Redis 的价值不是“快”这么简单，而是在数据库和应用之间放下一层可控、轻量、可建模的状态系统。

缓存 计数 分布式锁 排行榜
The Tension

项目刚开始，Redis 看起来只是“加个缓存”。流量一上来，它变成系统稳定性的考题。

命中率不稳

缓存策略不清晰，Redis 并不会自然提速，反而可能制造额外路径。

回源风暴

热点 key 失效、大量 key 集中过期、无效请求穿透，都可能把数据库打满。

边界被低估

分布式锁不是一个 setnx。过期、续期、释放、误删都在考验工程边界。

Mental Model

先把地图看清：Redis 不是替代数据库，而是补齐数据库前方的高频状态能力。

缓存 给热点数据加速，降低数据库压力。
计数器 承接点赞、浏览量、库存等高并发统计。
排行榜 用 Sorted Set 做快速排序和 Top N。
分布式锁 跨服务互斥，控制全局资源访问。
共享状态 多个服务实例之间共享轻量、短期状态。
Why Fast

Redis 的快，不只是“内存比磁盘快”。它是一整套面向高频读写的取舍。

Memory

基于内存操作，避开大部分磁盘访问成本。

Simple

数据结构直接，命令语义贴近业务建模。

Single

单线程执行模型减少锁竞争和上下文切换。

Network

网络模型和实现细节为高吞吐服务。

数据库像大仓库，稳、全、重；Redis 像前台小货架，只放高频物品，拿取极快。

Cache Path

缓存真正的核心不是“查 Redis”，而是控制命中、回源和失效后的秩序。

1

用户请求

请求进入应用服务，先判断是否适合走缓存路径。

2

查 Redis

命中则直接返回，把数据库从热点读中解放出来。

3

未命中

缓存没有结果时，进入回源逻辑。

4

查数据库

取到权威数据，同时处理空值、异常和边界。

5

写回缓存

设置合理 TTL，避免下一次相同请求重复打到数据库。

Data Structures

Redis 不只是 Key-Value。它让你用更接近业务语义的方式建模。

String简单缓存、计数器、库存、浏览量。
Hash用户会话、对象字段、轻量资料。
Set点赞用户、去重集合、交并差关系。
Sorted Set积分榜、热门文章、销量排行。

Cache Risk Radar

缓存系统最危险的地方，是它出问题时会把压力重新推回数据库。

穿透、击穿、雪崩不是三个面试词，而是三种真实的流量失控方式。

Risk Playbook

三类缓存风险，要用三套治理动作，不要一把 TTL 打天下。

问题图谱

穿透缓存和数据库都没有，恶意或异常请求持续落库。
击穿单个热点 key 失效，大量请求同一时刻回源。
雪崩大量 key 同时失效，数据库面对大面积流量。

解决动作

缓存空值 / 参数校验 / 布隆过滤器先挡住“不存在”的请求。
逻辑过期 / 热点永不过期 / 互斥重建让热点重建有秩序。
随机 TTL / 多级缓存 / 限流降级把集中失效打散。
Distributed Lock

分布式锁最难的不是“抢到锁”，而是锁在异常世界里仍然安全。

Real Scenes

商品详情、库存、热门排行，是 Redis 最容易产生价值的地方。

这些场景共同特征很清楚：读多写少、热点明显、短时间内允许一定程度不一致，或需要极快的排序和计数。

Misconceptions

初学者最容易误判的四件事，都会在真实项目里放大。

Redis 就是缓存

缓存只是入口。它还是数据结构服务、共享状态层、轻量协调组件。

加缓存一定提速

命中率、回源策略、失效策略没有设计好，缓存可能只是多了一段复杂链路。

分布式锁很简单

锁过期、锁续期、锁释放、异常边界，才是生产级设计的核心。

不区分一致性要求

有些数据可以短暂不一致，有些数据绝对不能。业务容忍度决定缓存策略。

Interview Snapshot

如果只能带走一张面试速记卡，就带走这张。

数据结构

String / Hash / List / Set / Sorted Set。记住它们对应的业务语义，而不是只背命令。

三大缓存问题

穿透查不存在，击穿打热点，雪崩是大量 key 同时失效。

锁为什么难

不是设置 key，而是保证过期、释放、安全性和异常边界都可控。

Closing

Redis 的价值，在性能与状态管理的交汇处。

真正有用的不是“会用 Redis”，而是知道它在系统里承担什么角色、适合什么场景、会带来哪些边界和风险。

减轻数据库压力 提升高频访问效率 管理轻量共享状态 警惕缓存系统性风险
