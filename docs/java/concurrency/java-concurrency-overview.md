# Java 并发编程总览：从线程到 AQS，为什么它总是既重要又容易学乱

Java 并发编程几乎是每个后端开发者都会遇到、但又很少有人真正学扎实的一块内容。你可以不天天手写线程池，也可以暂时不碰底层同步器，但你很难完全避开并发问题。接口并发访问、缓存竞争、异步任务、消息消费、批处理调度、线程池配置，这些场景本质上都在和并发打交道。

很多人学并发会陷入两个极端。一个极端是只背概念：线程、锁、`volatile`、`AQS`、CAS，背了很多词，写代码时还是一脸模糊。另一个极端是只会调 API：线程池能用，`synchronized` 能写，线上一出问题就不知道为什么吞吐量掉了、为什么死锁了、为什么线程池堆满了。

这篇文章想做的，不是一次性讲完并发所有细节，而是帮你建立一条更稳定的学习主线：并发编程到底在解决什么问题、有哪些核心模型、不同工具各自适合什么场景，以及为什么很多所谓“高阶问题”其实最终还是回到基础原理。

## 为什么并发编程对 Java 后端特别重要

Java 在企业应用和服务端领域长期占据重要地位，而服务端程序天然就运行在并发环境里。一个 Web 服务不会只处理一个请求；一个订单系统也不会只在单线程里跑；一个消息消费者往往也要在高并发环境下尽可能快地处理任务。

所以并发编程的重要性，首先来自现实环境，而不是来自面试。

你可以把它理解成：只要你的程序需要同时处理多个任务，就绕不开并发。而只要绕不开并发，就一定要处理下面这些核心问题：

- 多个线程如何协作
- 多个线程如何共享数据
- 数据共享时如何避免错误
- 如何在正确性和性能之间做平衡

从这个角度看，并发编程不是单独的一门“高级语法”，它其实是程序从“只要能跑”走向“能稳定跑”的关键一步。

## 并发编程到底在解决什么

很多初学者一上来就去背线程 API，结果越学越乱。其实并发编程真正要解决的问题，核心就三个：

- 可见性
- 原子性
- 有序性

### 可见性

一个线程修改了共享变量，另一个线程能不能及时看到？

如果不能，就会出现“明明改了，别人还像没改一样”的问题。

### 原子性

一个操作是不是不可分割？

比如 `count++` 看起来是一行代码，但底层并不是原子操作，它通常会分成读取、计算、写回三个步骤。多个线程同时执行时，就可能出现结果不符合预期的情况。

### 有序性

程序写出来的顺序，和 CPU 实际执行的顺序，是否完全一致？

为了优化性能，编译器和处理器可能会做指令重排。如果代码依赖某种执行顺序，而你没有建立正确的并发约束，就会出现非常隐蔽的问题。

你如果能把这三个问题真正理解透，后面再学 `synchronized`、`volatile`、Lock、CAS、线程池时，很多东西都会顺很多。

## 从线程开始理解并发

并发编程最表层的实体是线程。线程是 CPU 调度的基本单位，它让程序可以同时执行多条任务路径。

但很多人学线程时容易停留在“怎么创建线程”。实际上，创建线程本身并不难，难的是如何管理线程和线程间共享状态。

例如：

```java
public class CounterDemo {
    private static int count = 0;

    public static void main(String[] args) throws InterruptedException {
        Thread t1 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) {
                count++;
            }
        });

        Thread t2 = new Thread(() -> {
            for (int i = 0; i < 10000; i++) {
                count++;
            }
        });

        t1.start();
        t2.start();
        t1.join();
        t2.join();

        System.out.println(count);
    }
}
```

很多初学者第一次看到这段代码时，会以为输出一定是 `20000`。但实际上结果经常小于这个值。原因不在于“线程不稳定”，而在于 `count++` 不是原子操作。

这就是并发学习的一个分水岭：线程不是重点，共享数据的正确性才是重点。

## synchronized：最经典、也最容易被误解的关键字

只要学 Java 并发，几乎一定会最先接触 `synchronized`。它的作用可以简单理解成：在某一段代码执行时，先拿到某个监视器锁，让别的线程暂时不能同时执行受保护区域。

例如：

```java
public class SafeCounter {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int getCount() {
        return count;
    }
}
```

这段代码的好处是简单直接，能同时处理原子性和可见性问题。你不需要自己设计复杂同步逻辑，就能先把共享数据保护起来。

但 `synchronized` 也很容易被误解：

- 它不是“用了就一定高性能”
- 它不是“哪里不安全就到处加”
- 它也不是只要锁住代码，业务逻辑就一定正确

真正重要的是：你要清楚你在保护什么共享状态，锁的粒度是否合适，以及有没有更适合的并发模型。

## volatile：它能解决什么，不能解决什么

`volatile` 经常和 `synchronized` 一起出现，但它们解决的问题并不一样。

`volatile` 的核心价值是：

- 保证可见性
- 在一定程度上约束重排序

但它**不保证复合操作的原子性**。

例如下面这段代码：

```java
public class VolatileDemo {
    private volatile int count = 0;

    public void increment() {
        count++;
    }
}
```

即使 `count` 被声明为 `volatile`，多个线程同时执行 `increment()` 时，结果仍然可能不正确。因为 `count++` 本身仍然是多步操作。

所以你可以把 `volatile` 理解成一种更轻量的并发语义工具。它适合表示某个状态“改了之后，别的线程要尽快看到”，比如停止标志、配置刷新标志等。但它不适合拿来替代锁处理复杂共享更新。

## Lock 和 AQS：为什么 Java 还要提供一套锁体系

既然有了 `synchronized`，为什么还需要 `Lock`？

答案是：并发控制不只是“能锁住就行”，很多场景还需要更灵活的能力，比如：

- 尝试获取锁
- 可中断地获取锁
- 公平锁与非公平锁
- 多条件队列

这时候 `ReentrantLock` 等显式锁就显得更灵活。

而再往底层看，你会遇到 `AQS`，也就是 `AbstractQueuedSynchronizer`。很多 Java 并发工具都建立在它之上，比如：

- `ReentrantLock`
- `Semaphore`
- `CountDownLatch`
- `ReentrantReadWriteLock`

如果说 `synchronized` 是会用就能解决不少问题，那么理解 `AQS` 会让你真正看懂“Java 并发工具到底是怎么组织起来的”。

不过对于大多数开发者来说，学习顺序不应该是一上来就啃 `AQS` 源码，而应该是：

1. 先理解共享状态问题
2. 再理解锁和线程协作模型
3. 再用这些视角去看 `AQS`

这样你读源码时才知道它在解决什么，而不是被细节淹没。

## 线程池：并发编程里最容易“会用但不会配”的部分

Java 项目里比“手写线程”更常见的，是线程池。因为线程创建和销毁成本不低，如果每来一个任务都新建线程，系统很快就会变得不可控。

线程池的价值在于：

- 复用线程
- 控制并发度
- 平衡资源和吞吐量
- 提供队列和拒绝策略

但线程池也是线上问题高发区。很多人只是会写：

```java
ExecutorService executor = Executors.newFixedThreadPool(10);
```

却不清楚：

- 线程数为什么是 10
- 队列多大合适
- 任务堆积时怎么办
- 拒绝策略意味着什么

如果这些问题没有想清楚，线程池就可能从“性能优化工具”变成“问题放大器”。

所以学线程池，不能只记参数名字，更重要的是建立一个系统视角：任务的生产速度、消费速度、资源上限和失败策略之间是什么关系。

## 并发容器：不是“线程安全版集合”这么简单

在多线程环境下，直接使用普通的 `HashMap`、`ArrayList`、`HashSet` 往往会有问题，这时你会接触到并发容器，比如：

- `ConcurrentHashMap`
- `CopyOnWriteArrayList`
- `BlockingQueue`

这些容器并不是简单给普通集合“加锁”而已，它们背后体现了不同场景下的权衡。

例如：

- `ConcurrentHashMap` 更适合高并发读写
- `CopyOnWriteArrayList` 更适合读多写少
- `BlockingQueue` 适合生产者消费者模型

如果你不知道这些容器的设计意图，很容易用错工具。比如在高频写场景下滥用 `CopyOnWriteArrayList`，性能会很差；在需要阻塞协作的场景里用普通队列，又会导致大量自己补同步逻辑。

## 一个常见的真实场景：异步处理为什么总是容易出问题

假设你有一个需求：用户下单后，异步发送短信和站内通知。

很多人第一反应是：

```java
new Thread(() -> sendSms(order)).start();
new Thread(() -> sendMessage(order)).start();
```

这能跑，但很快就会暴露出问题：

- 线程不可控，资源难管理
- 异常不容易统一处理
- 任务量大时系统会被拖垮

更合理的做法通常是使用线程池：

```java
executorService.submit(() -> sendSms(order));
executorService.submit(() -> sendMessage(order));
```

再进一步，如果任务间耦合复杂、失败重试要求高，还可能需要消息队列。这个例子很能说明一件事：并发编程不只是 API 使用问题，它最终会影响系统设计方式。

## 学并发时最常见的误区

### 误区一：把并发等同于多线程 API

真正难的不是 `Thread`、`Runnable`、`ExecutorService` 这些名字，而是共享状态、同步语义和系统行为。

### 误区二：一学并发就直冲源码

源码当然值得看，但如果没有“为什么需要这个工具”的问题意识，源码很容易把你劝退。

### 误区三：只追求高性能，不先保证正确性

并发里最危险的不是慢，而是错。数据错乱、死锁、线程泄漏、任务丢失，这些问题往往比性能差更难处理。

### 误区四：线程池参数凭感觉配

很多线上问题不是不会用线程池，而是参数没有结合任务特征、机器资源和流量模型去设计。

## 一条更稳的 Java 并发学习路线

如果你想真正把并发学扎实，我建议按下面顺序来：

1. 线程和共享变量问题
2. Java 内存模型基础
3. `synchronized` 与 `volatile`
4. Lock、CAS、原子类
5. 线程池
6. 并发容器
7. `AQS` 与常见同步器

这个顺序的重点是：先建立问题意识，再学工具，再看底层原理。

## 总结

Java 并发编程之所以难，不是因为概念多，而是因为它同时要求你理解程序行为、共享状态、工具语义和系统资源。你只学语法，会停留在表面；你只看源码，又可能因为缺少问题背景而越学越乱。

真正好的学习方式，是先抓住可见性、原子性、有序性这三个核心问题，再逐步理解 `synchronized`、`volatile`、Lock、线程池、并发容器和 `AQS` 各自在解决什么。

当你开始用“这段并发逻辑到底保护了什么状态、在什么边界上协作、会不会在流量上升时失控”去思考问题时，你才算真正走进了 Java 并发的世界。
