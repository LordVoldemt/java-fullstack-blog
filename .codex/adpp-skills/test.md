# ADPP Skill: 测试 (test)

## 触发条件
用户运行 `/adpp:test` 或描述"测试"、"质检"、"测试用例"、"QA"时激活。

---

## 前置检查

```
必须存在：
  ✅ .adpp/changes/*/tasks.md（任务已完成）
  ✅ 对应的代码实现
```

---

## 强制流程

```
STEP 1: 读取实现清单，确认测试范围
STEP 2: 补全测试用例（超出 TDD 已覆盖的）
STEP 3: 运行完整测试套件
STEP 4: 安全扫描
STEP 5: 生成测试报告
STEP 6: ▶ [人工门控] 测试报告确认
```

---

## STEP 1：确认测试范围

如果当前变更存在 `.adpp/changes/{{ feature-dir }}/ux-notes.md`，且涉及前端界面：

- 读取其中的交互流程
- 读取组件清单
- 读取响应式要求
- 如设计稿路径在当前环境可访问，可将其作为 E2E / 视觉回归 / 响应式测试补充输入

如果设计稿路径不可访问，则以 `ux-notes.md` 的文字描述为准，不得假装已按设计稿完成验证。

```
📋 测试范围分析：

已有测试（TDD 阶段生成）：
  {{ 列出已有测试文件 }}

需要补充的测试类型：
  □ 集成测试（跨模块/跨服务）
  □ 性能测试（关键接口）
  □ 安全测试（OWASP Top10 相关）
  □ 边界值测试（未被单元测试覆盖的）

确认测试范围？
```

---

## STEP 2：补全测试用例

### 测试分层策略

```
单元测试（Unit）：
  - 覆盖率目标：≥ 80%
  - 每个函数/方法独立测试
  - Mock 所有外部依赖
  - 由 Claude 生成

集成测试（Integration）：
  - 覆盖核心业务链路 100%
  - API 接口测试（真实数据库/Mock 外部 API）
  - 由 Claude 生成

E2E 测试（End-to-End）：
  - 覆盖主要用户流程
  - 从用户视角验证
  - 建议使用 Playwright / Cypress

跨模型质检（Cross-Model Review）：
  - 使用不同模型（如 Gemini）对代码进行独立 Review
  - 重点：找出 Claude 可能忽略的问题
```

### 补全测试的重点场景

```
必须覆盖的测试类型：

1. 正常流程（Happy Path）
   - 输入合法数据，验证预期输出

2. 边界值（Boundary Values）
   - 空字符串、null、0、最大值、最小值
   - 数组：空数组、单元素、最大长度

3. 异常处理（Error Cases）
   - 无效输入的错误码
   - 外部服务不可用时的降级
   - 并发/竞态条件

4. 业务规则验证
   - PRD 中每条验收标准对应至少一个测试

5. 安全测试
   - SQL 注入尝试
   - XSS 尝试（前端）
   - 未授权访问
   - 越权操作
```

## STEP 3：运行完整测试套件

```bash
# AI 生成并运行测试命令（基于技术栈）

# Python
pytest --cov={{ module }} --cov-report=html -v

# Node.js
npm test -- --coverage

# Go
go test ./... -cover

# Java
./mvnw test jacoco:report
```

测试结果展示格式：
```
📊 测试结果：

单元测试：  {{ passed }}/{{ total }} 通过
集成测试：  {{ passed }}/{{ total }} 通过
测试覆盖率：{{ coverage }}%（目标：≥80%）

失败的测试：
  ❌ {{ test_name }}：{{ 失败原因 }}

覆盖率不足的模块：
  ⚠️ {{ module }}：{{ N }}%（需要补充测试）
```

## STEP 4：安全扫描

基于技术栈选择扫描方式：

```
Python:     bandit -r {{ src_dir }}
Node.js:    npm audit + eslint-plugin-security
Go:         gosec ./...
Java:       OWASP dependency-check
通用:       检查 .adpp/config.yaml 中的 danger_operations.patterns
```

OWASP Top10 检查清单：
```
□ A01 访问控制失效 → 验证所有接口的鉴权
□ A02 加密失败 → 检查敏感数据是否明文存储/传输
□ A03 注入 → SQL/命令注入防护
□ A04 不安全设计 → 对照设计文档
□ A05 安全配置错误 → 检查默认密码、不必要的端口
□ A06 易受攻击的组件 → npm audit / pip-audit
□ A07 身份认证失败 → Token 有效期、会话管理
□ A08 数据完整性失败 → 反序列化安全
□ A09 日志记录不足 → 关键操作是否有审计日志
□ A10 SSRF → 外部请求是否有白名单
```

## STEP 6：人工门控 — 测试报告确认

**必须暂停**：

```
▶ [测试确认] 测试报告已生成：.adpp/changes/{{ feature-dir }}/test-report.md

关键指标：
  测试覆盖率：{{ N }}% {{ ≥80% ? "✅" : "⚠️ 未达标" }}
  高危安全问题：{{ N }} 个 {{ ==0 ? "✅" : "❌ 需处理" }}
  失败测试：{{ N }} 个 {{ ==0 ? "✅" : "❌ 需修复" }}

PRD 验收标准覆盖：{{ N }}/{{ total }}

{{ 如果有不达标项 }}：
  ⚠️ 以下问题需要处理后才建议进入部署：
  - {{ 问题列表 }}

确认后输入：
  "确认测试" → 进入部署阶段，或在无需部署时运行 /adpp:closeout
  "修复：xxx" → 修复指定问题后重新测试
  "豁免：xxx（原因）" → 标记为已知问题，接受风险继续
```

**高危安全问题不得豁免，必须修复。**

---

## 禁止行为

| 禁止 | 原因 |
|---|---|
| 声称测试通过而不实际运行 | 同 implement skill |
| 覆盖率不足就进入下一阶段 | 质量门控不可妥协 |
| 跳过安全扫描 | 安全是底线 |
| 高危安全问题豁免 | 绝对禁止 |
| 测试报告未确认就部署 | 人工门控不可跳过 |
