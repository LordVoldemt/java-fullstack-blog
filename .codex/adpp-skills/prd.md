# ADPP Skill: 需求分析 (prd)

## 触发条件
用户运行 `/adpp:prd` 或描述新功能需求、提到"PRD"、"需求文档"、"功能设计"时激活。

---

## 核心原则

PRD 是整个项目的**单一事实来源**。
- 所有代码和 AI 行为的对错，以 PRD 为准
- 任何需求变更，必须先修改 PRD，再修改代码
- PRD 经人工审核确认后才能进入设计阶段

---

## 强制流程

```
STEP 1: 读取项目上下文
STEP 2: Socratic 澄清问答
STEP 3: 生成 PRD 草稿
STEP 4: ▶ [人工门控] PRD 审核
STEP 5: PRD 归档
```

---

## STEP 1：读取项目上下文

执行前必须读取：
- `.adpp/config.yaml` → 技术栈、团队规模
- `.adpp/spec/` → 相关规范
- `.adpp/changes/` → 已有变更，避免重复

如果 `.adpp/config.yaml` 不存在，停止并提示：
```
❌ 未找到 ADPP 配置。请先运行 /adpp:init 初始化项目。
```

---

## STEP 2：Socratic 澄清问答

**规则：每次只问一个问题，等待用户回答后再问下一个。**

必问清单（按顺序）：
```
Q1: 这个功能/需求用一句话描述是什么？
Q2: 主要用户是谁？他们的核心诉求是什么？
Q3: 功能边界在哪里？哪些是必须有的，哪些是不在本次范围内的？
Q4: 有没有需要集成的外部系统或依赖？
Q5: 验收标准是什么？怎么算做完了？
Q6: 有没有非功能需求？（性能指标、安全要求、兼容性等）
```

遇到模糊回答时，追问一层：
- "你说的 XX 具体是指...？"
- "这里有两种理解方式：A 还是 B？"

**不得一次把 6 个问题全部列出来。**

---

## STEP 3：生成 PRD 草稿

变更目录命名规则：`.adpp/changes/{{ YYYY-MM-DD }}-{{ feature-slug }}/`

生成以下文件：

### prd.md 完整格式

```markdown
# PRD: {{ 功能名称 }}

**版本**: v1.0  
**日期**: {{ date }}  
**状态**: 草稿 / 待审核 / 已确认 / 已归档  
**作者**: {{ user }}

---

## 1. 背景与目标

### 1.1 背景
{{ 为什么要做这个功能，解决什么问题 }}

### 1.2 目标
{{ 本次迭代要达成的具体目标，可量化 }}

### 1.3 非目标（Out of Scope）
{{ 明确列出不在本次范围内的内容 }}

---

## 2. 用户故事

| 角色 | 希望 | 以便 |
|---|---|---|
| 作为 {{ 用户类型 }} | 我希望 {{ 功能描述 }} | 以便 {{ 业务价值 }} |

---

## 3. 功能需求

### 3.1 {{ 模块名称 }}

**描述**: {{ 功能描述 }}

**交互流程**:
1. 用户 {{ 操作 }}
2. 系统 {{ 响应 }}
3. {{ 异常情况处理 }}

**数据需求**: {{ 涉及的数据字段 }}

### 3.2 {{ 模块名称 }}
...（按功能模块拆分，每个模块独立小节）

---

## 4. 非功能需求

| 类型 | 要求 |
|---|---|
| 性能 | {{ 响应时间、并发量等 }} |
| 安全 | {{ 鉴权、数据安全要求 }} |
| 兼容性 | {{ 浏览器、OS、API 版本等 }} |
| 可用性 | {{ SLA 要求 }} |

---

## 5. 验收标准

- [ ] {{ 可验证的验收条件 1 }}
- [ ] {{ 可验证的验收条件 2 }}
- [ ] {{ 边界情况已覆盖 }}

---

## 6. 依赖与风险

| 依赖/风险 | 说明 | 应对措施 |
|---|---|---|
| {{ 外部依赖 }} | {{ 说明 }} | {{ 措施 }} |

---

## 7. 变更历史

| 版本 | 日期 | 变更内容 |
|---|---|---|
| v1.0 | {{ date }} | 初始版本 |
```

---

## STEP 4：人工门控 — PRD 审核

生成 PRD 后，**必须暂停并输出**：

```
▶ [人工确认] PRD 草稿已生成：.adpp/changes/{{ feature-dir }}/prd.md

请检查以下关键点：
  □ 功能边界是否清晰？Out of Scope 是否明确？
  □ 验收标准是否可验证、可量化？
  □ 非功能需求是否遗漏？
  □ 用户故事是否覆盖主要场景？

确认无误后输入：
  "确认 PRD" → 进入系统设计阶段
  "修改：xxx" → 描述需要修改的内容，我来更新
  "重新澄清" → 重新进行问答
```

**在用户输入"确认 PRD"之前，不得生成任何设计或代码。**

---

## STEP 5：PRD 归档

用户确认后：
1. 更新 PRD 文件状态为 `已确认`
2. 在变更目录创建 `CHANGELOG.md`，记录本次 PRD 确认
3. 更新 `.adpp/workspace/current-phase.md`：
   ```markdown
   当前阶段: 需求已确认，等待进入系统设计
   当前变更: .adpp/changes/{{ feature-dir }}/
   PRD 状态: ✅ 已确认
   下一步: 运行 /adpp:design
   ```
4. 输出：
   ```
   ✅ PRD 已确认归档。
   运行 /adpp:design 开始系统设计。
   ```

---

## 需求变更处理

如果用户在设计/编码阶段提出需求变更：

```
⚠️ 检测到需求变更。根据 ADPP 规范，需要先更新 PRD。

受影响的 PRD：.adpp/changes/{{ feature-dir }}/prd.md
建议修改内容：{{ AI 分析受影响的章节 }}

是否现在更新 PRD？
  "是" → 更新 PRD，重新确认后继续
  "否" → 暂不处理，继续当前任务（风险：代码和 PRD 会不一致）
```

---

## 禁止行为

| 禁止 | 原因 |
|---|---|
| 跳过澄清问答直接生成 PRD | 需求模糊是所有问题的根源 |
| 一次问多个问题 | 降低用户思考质量 |
| PRD 未确认就开始设计 | 门控节点不可跳过 |
| 需求变更时不更新 PRD | 导致代码和文档不一致 |
| 验收标准写成主观描述 | "界面美观"不是验收标准 |
