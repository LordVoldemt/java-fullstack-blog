# 编码规范 — {{ project_name }}

> 本规范为 ADPP 内置默认模板，适合作为团队基线。初始化后可按项目技术栈和存量约束做增删，但不建议删除核心质量门槛。
>
> 对齐参考：
> - Conventional Commits 1.0.0
> - Semantic Versioning 2.0.0
> - Twelve-Factor App
> - OWASP ASVS / OWASP Top 10

---

## 1. 目标

- 让代码具备一致性、可维护性、可测试性和可审查性
- 为 AI 生成代码提供明确边界，减少风格漂移和隐性风险
- 降低多人协作、跨团队交接和后续重构成本

---

## 2. 基本原则

- **可读性优先于聪明写法**
- **显式优先于隐式**
- **组合优先于耦合**
- **测试与代码同等重要**
- **安全和可观测性默认开启**

---

## 3. 目录与模块边界

### 推荐目录分层

```text
src/
  api/            # 接口层 / 路由 / controller
  application/    # 用例编排 / service / command
  domain/         # 核心领域模型与规则
  infrastructure/ # DB / MQ / cache / third-party adapters
  shared/         # 通用工具、基础设施封装

tests/
  unit/
  integration/
  contract/
  e2e/
```

### 模块边界要求

- `api` 不直接操作数据库细节
- `domain` 不依赖 Web 框架、ORM 或页面框架
- `infrastructure` 可以依赖外部 SDK，但不得反向污染上层抽象
- 公共能力进入 `shared` 前，至少被两个模块复用且职责清晰

---

## 4. 命名规范

### 通用规则

- 变量、函数、类、模块命名必须表达业务语义，避免 `data`, `obj`, `tmp`, `handler2`
- 布尔变量使用可读谓词：`isEnabled`, `hasAccess`, `shouldRetry`
- 函数名用动作开头：`createUser`, `syncProjectMembers`, `calculateQuota`
- 常量统一用语义化命名，不使用裸值散落在代码里

### Python

```text
变量/函数：snake_case
类名：PascalCase
常量：UPPER_SNAKE_CASE
模块名：snake_case.py
私有方法：_leading_underscore
```

### TypeScript / JavaScript

```text
变量/函数：camelCase
类名/类型：PascalCase
常量：UPPER_SNAKE_CASE
React 组件：PascalCase
文件名：与团队栈保持一致，默认 kebab-case 或 camelCase 二选一，禁止混用
```

## 5. 函数与复杂度控制

- 单个函数优先控制在 **30 行以内**
- 参数数量优先控制在 **4 个以内**
- 嵌套层级优先控制在 **3 层以内**
- 单个文件优先控制在 **300 行以内**；超出时考虑拆分
- 复杂逻辑必须拆成命名清晰的小函数，不用长注释掩盖复杂度

### 禁止行为

- 超长函数混合校验、编排、持久化、日志和异常处理
- 为了“少写几行”把多个职责硬塞进三元表达式或链式调用
- 用复制粘贴代替抽象，但抽象前先确认存在稳定复用

---

## 6. 类型、契约与接口

- 新代码优先使用显式类型声明
- 对外接口、DTO、Schema、API 响应格式必须稳定且可验证
- 公共函数必须明确输入、输出、异常或错误返回语义
- 禁止返回“靠调用者猜”的混合结构
- 对外部依赖的响应做适配层，避免外部模型直接渗透到核心域模型

---

## 7. 错误处理

- 禁止吞异常，如 `except: pass`、空 `catch`
- 错误信息必须包含足够上下文，但不得泄露密钥、口令、PII
- 业务异常与系统异常分层处理
- API 错误码和错误结构必须稳定，不随实现细节漂移

### 推荐模式

- `400`：参数或前置条件不满足
- `401/403`：认证与鉴权失败
- `404`：资源不存在
- `409`：并发冲突或状态冲突
- `422`：业务规则校验失败
- `5xx`：系统异常或下游不可用

---

## 8. 日志与可观测性

- 关键业务操作必须记录结构化日志
- 外部请求必须记录：目标、耗时、结果、重试次数
- 错误日志必须带请求上下文、租户/用户标识、trace id 或 request id
- 默认不在生产日志中输出敏感字段原文

### 至少需要记录的事件

- 登录、登出、权限拒绝
- 创建、更新、删除
- 外部系统调用失败
- 任务调度开始/结束/失败
- 部署、迁移、危险操作确认

---

## 9. 配置与密钥

- 配置来自环境变量、密钥管理或受控配置文件
- 禁止硬编码 API Key、密码、Token、连接串
- 每个配置项需说明默认值、来源和作用范围
- 环境差异通过配置解决，不通过 `if env == ...` 散落在业务代码

---

## 10. 数据库与持久化

- Schema 变更必须通过迁移脚本，不直接手工改线上库
- 读写逻辑分离，避免在 controller 中拼 SQL
- 所有删除或批量更新操作必须显式限制条件
- 查询需考虑索引、分页、N+1 与事务边界

### 禁止行为

- `SELECT *`
- 无 `WHERE` 的批量更新/删除
- 将 ORM 模型直接暴露为 API 响应
- 在循环内重复发起数据库或外部服务调用

---

## 11. 安全基线

- 所有输入默认不可信，必须做校验与编码/转义
- 默认开启鉴权与最小权限
- 文件上传、命令执行、模板渲染、反序列化属于高风险区，必须显式审查
- 对外访问必须设置超时、重试和失败兜底
- 依赖升级需关注已知漏洞与许可证风险

### 高风险检查项

- SQL 注入
- XSS / 模板注入
- SSRF
- IDOR / 越权访问
- 敏感日志泄露
- 未受控的文件系统操作

---

## 12. 注释与文档字符串

- 注释用于解释“为什么”，而不是复述“做了什么”
- 复杂公共函数、关键领域规则、绕过历史兼容问题的代码必须写注释
- 公共接口、SDK、工具函数需要文档字符串或等价说明

### 推荐包含内容

- 输入约束
- 返回值含义
- 可能抛出的异常或错误码
- 边界场景

---

## 13. Code Review 检查表

- 是否符合 PRD / design / tasks 约束
- 是否有测试覆盖关键路径和错误路径
- 是否引入安全、并发、数据一致性风险
- 是否存在过度设计或不必要抽象
- 是否有清晰的日志、错误处理和回滚路径

---

## 14. Git、版本与提交规范

### 提交规范

采用 **Conventional Commits**：

```text
feat: 新功能
fix: Bug 修复
refactor: 重构但不改变外部行为
test: 测试相关
docs: 文档变更
chore: 工具、构建、依赖、脚手架
perf: 性能优化
build: 构建系统或依赖打包
ci: CI/CD 流程
```

### 提交要求

- 一次提交只做一类相对单一的变更
- 提交信息写“意图”和“影响面”，不要只写 `update`、`fix bug`
- 发布版本遵循 **Semantic Versioning**

---

## 15. AI 生成代码附加约束

- AI 不得跳过测试、门控和文档同步
- AI 生成代码必须遵守本规范，而不是“生成后再解释”
- AI 不能擅自引入新的架构模式、基础设施依赖或高风险库
- AI 如需突破规范，必须明确说明原因并等待人工确认
