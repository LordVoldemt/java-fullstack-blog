# 文档规范 — {{ project_name }}

> 本模板为 ADPP 内置默认文档标准，用于统一项目文档结构、版本方式与追溯规则。
>
> 对齐参考：
> - Keep a Changelog 1.1.0
> - Semantic Versioning 2.0.0
> - ADR（Architecture Decision Record）通用实践

---

## 1. 文档目标

- 所有关键决策可被后来者理解、复盘和继承
- 文档服务开发、测试、部署与交接，而不是仅供汇报
- 文档与代码保持同轮更新，不允许长期漂移

---

## 2. 文档分层

### 2.1 Source of Truth

这些文档定义事实边界，变更时必须优先更新：

- `PRD`：需求目标、边界、验收标准
- `design`：架构方案、接口、数据模型、关键技术决策
- `tasks`：实施拆解、状态、依赖关系
- `API/OpenAPI`：接口契约与字段语义
- `Runbook / Deploy`：部署与运行说明

### 2.2 Derived Docs

这些文档由事实源派生，可在同轮同步更新：

- README
- 用户使用手册
- 演示材料
- QA Matrix
- 变更摘要 / 发布说明

### 2.3 Evidence Docs

用于记录本轮真实执行证据：

- test-report
- deploy-record
- closeout-note
- progress note / review note

---

## 3. 文档状态

统一使用以下状态：

- `草稿`
- `待评审`
- `已确认`
- `已发布`
- `已归档`
- `已废弃`

任何文档的状态切换都应有时间、责任人或上下文说明。

---

## 4. 每类核心文档最少结构

### 4.1 PRD

- 背景与目标
- 用户角色 / 用户故事
- 范围与非范围
- 功能需求
- 非功能需求
- 验收标准
- 风险与待确认项

### 4.2 设计文档

- 架构图 / 模块边界
- 技术选型与权衡
- API / 数据模型
- 安全、性能、扩展性考虑
- 迁移与兼容方案
- 风险与回滚思路

### 4.3 任务拆解

- 任务编号
- 任务描述
- 负责人 / 执行者
- 依赖项
- 状态
- 验证方式

### 4.4 Runbook / Deploy

- 环境前置条件
- 部署步骤
- 回滚步骤
- 健康检查
- 风险说明
- 常见故障排查

## 5. Markdown 与排版规范

- 一个文档只表达一个主主题
- 标题层级不超过四级
- 表格用于稳定结构信息，长解释优先使用短段落与扁平列表
- 命令、路径、环境变量统一使用代码格式
- 时间统一使用绝对日期，如 `2026-04-21`

### 文件命名建议

- 面向阶段文档：`prd.md`、`design.md`、`tasks.md`
- 面向轮次事实：`YYYY-MM-DD-topic.md`
- 面向规范/参考：`kebab-case.md`

---

## 6. 变更日志与版本管理

- 版本号遵循 **Semantic Versioning**
- 变更日志推荐遵循 **Keep a Changelog**
- 每次发布记录建议至少包含：
  - Added
  - Changed
  - Fixed
  - Deprecated
  - Removed
  - Security

### 最小 changelog 条目

- 版本号
- 日期
- 变更摘要
- 破坏性变更说明
- 升级/回滚注意事项

---

## 7. 决策记录（ADR）

下列情况建议新增 ADR：

- 更换核心技术栈
- 引入新存储、新消息系统或新身份体系
- 采用新的架构模式
- 引入会长期影响维护成本的工程规范

### ADR 建议结构

- 背景
- 决策内容
- 备选方案
- 权衡与影响
- 生效日期
- 关联 PRD / 设计文档

---

## 8. 可追溯性要求

每次正式交付至少能够追溯：

- PRD -> design
- design -> tasks
- tasks -> code / commit / MR
- tasks -> tests
- closeout -> evidence

任何一个环节缺失，都会降低后续审查、交接与维护效率。

---

## 9. API 文档要求

- 对外接口必须描述请求、响应、错误码和鉴权方式
- 字段含义要稳定，不使用模糊名称
- 示例请求和响应优先与真实对象一致
- Breaking Change 必须在 changelog 和 closeout 中明确标记

---

## 10. closeout 与项目文档同步

- 本轮改动完成后必须执行收口
- 不允许只更新 progress note 而不更新事实源文档
- closeout note 需要写明：
  - 本轮范围
  - 更新了哪些 source docs
  - 更新了哪些 derived docs
  - 跑了哪些验证
  - 证据在哪里

---

## 11. 文档质量检查表

- 文档是否回答了“为什么做、做了什么、怎么验证、有什么风险”
- 是否存在历史说法与当前实现冲突
- 是否存在未标记状态的临时草稿
- 是否有可执行命令、示例或图示支撑关键说明
- 是否有足够信息让新加入成员在不口头交接的情况下理解背景
