# ADPP Skill: 系统设计 (design)

## 触发条件
用户运行 `/adpp:design` 或描述"系统设计"、"架构设计"、"数据库设计"、"技术方案"时激活。

---

## 前置检查（必须通过才能继续）

```javascript
// 伪代码：每次执行前检查
if (!exists(".adpp/changes/*/prd.md 且状态=已确认")) {
  停止并提示: "❌ 未找到已确认的 PRD。请先运行 /adpp:prd 完成需求分析。"
}
```

读取：
- `.adpp/changes/{{ current-feature }}/prd.md` → 功能需求
- `.adpp/spec/coding-standards.md` → 编码规范约束
- `.adpp/config.yaml` → 技术栈、项目约束

---

## 强制流程

```
STEP 1: 分析 PRD，提取系统设计约束
STEP 2: 生成多方案系统设计
STEP 3: ▶ [人工门控] 技术评审
STEP 4: 判断是否进入 UX 阶段
STEP 5: （纯后端项目）生成细粒度 Plan（tasks.md）
STEP 6: 归档与阶段流转
```

---

## STEP 1：分析 PRD，提取设计约束

从 PRD 中提取：
- 功能模块列表
- 非功能需求（性能、安全、可扩展性）
- 外部依赖（第三方 API、数据库类型等）
- 验收标准（反向推导技术实现要求）
- 是否涉及前端界面、交互流程、组件或响应式要求

输出分析摘要：
```
📋 设计约束分析：
  功能模块：{{ N }} 个
  性能要求：{{ 响应时间等 }}
  安全要求：{{ 鉴权方式等 }}
  技术栈约束：{{ 来自 config.yaml }}
  前端界面影响：{{ 是/否 }}
  关键风险：{{ AI 识别的技术风险点 }}
```

## STEP 3：人工门控 — 技术评审

**必须暂停并输出**：

```
▶ [技术评审] 设计文档已生成：.adpp/changes/{{ feature-dir }}/design.md

技术评审检查清单（请逐项确认）：
  □ 架构设计是否符合项目整体技术方向？
  □ 数据库设计是否合理？索引是否遗漏？
  □ API 设计是否符合 RESTful/团队规范？
  □ 安全设计是否覆盖主要威胁？
  □ 技术风险是否可接受？
  □ 方案 A 还是方案 B？（如有对比）
  □ 是否涉及前端界面，需要进入 /adpp:ux 阶段？

确认后输入：
  "确认设计 [方案A/B]" → 系统设计确认；若涉及前端界面，则下一步进入 /adpp:ux
  "修改：xxx" → 描述修改点
  "需要讨论：xxx" → 标记为待讨论，暂不继续
```

**架构评审是技术人员价值最大化的节点，不可绕过。**

---

## STEP 4：判断是否进入 UX 阶段

技术评审确认后，按以下规则流转：

### 情况 A：当前变更涉及前端界面 / 交互 / 响应式

- 不在本阶段生成最终 `tasks.md`
- 更新阶段为“系统设计已确认，等待前端设计”
- 下一步：运行 `/adpp:ux`

### 情况 B：纯后端 / 纯接口 / 纯数据 / 纯脚本变更

- 直接在本阶段生成 `tasks.md`
- 下一步：运行 `/adpp:implement`

判定原则：

- 只要实现结果会直接改变页面结构、页面流程、组件行为、状态页或响应式表现，就应进入 `/adpp:ux`
- 如果只是接口、数据、脚本、部署或纯后端逻辑，不必新增 UX 阶段

## STEP 6：归档与阶段流转

### 若进入 UX 阶段

更新 `.adpp/workspace/current-phase.md`：

```markdown
当前阶段: 系统设计已确认，等待前端设计
当前变更: .adpp/changes/{{ feature-dir }}/
系统设计状态: ✅ 已确认（方案 {{ A/B }}）
下一步: 运行 /adpp:ux
```

### 若无需进入 UX 阶段

更新 `.adpp/workspace/current-phase.md`：

```markdown
当前阶段: 系统设计已确认，进入编码阶段
当前变更: .adpp/changes/{{ feature-dir }}/
系统设计状态: ✅ 已确认（方案 {{ A/B }}）
Plan 状态: ✅ 已生成（{{ N }} 个任务）
下一步: 运行 /adpp:implement
```

---

## 禁止行为

| 禁止 | 原因 |
|---|---|
| 没有 PRD 就开始设计 | 设计必须基于确认的需求 |
| 只给一个方案 | 技术评审需要对比选择 |
| 跳过技术评审 | 架构错误在编码后修复成本极高 |
| 在涉及前端界面的场景下跳过 `/adpp:ux` | 页面实现会退化成凭猜测编码 |
| tasks.md 任务粒度过大 | 单个任务 > 50 行难以 AI 并行执行 |
| 遗漏安全设计章节 | 安全是设计阶段必须考虑的 |
