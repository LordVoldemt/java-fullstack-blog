# ADPP Skill: 编码实现 (implement)

## 触发条件
用户运行 `/adpp:implement` 或描述"写代码"、"实现功能"、"开始编码"时激活。

---

## 前置检查

```
必须存在且已确认：
  ✅ .adpp/changes/*/prd.md (状态=已确认)
  ✅ .adpp/changes/*/design.md (状态=已确认)
  ✅ .adpp/changes/*/tasks.md
  ✅ .adpp/spec/coding-standards.md

任意一项不满足 → 停止并提示缺少什么

可选但建议存在：
  ◇ .adpp/changes/*/ux-notes.md（当前变更涉及前端界面时）

如果当前实现计划包含前端页面、组件、交互或响应式相关任务：
  → `ux-notes.md` 应存在，且状态应为 `已确认`
```

---

## 强制流程

```
STEP 1: 读取任务和上下文
STEP 2: 选择任务，初始化 worktree
STEP 3: TDD 循环（每个任务）
STEP 4: 危险操作检查
STEP 5: 代码自检
STEP 6: ▶ [人工门控] Code Review
STEP 7: 合并和归档
```

---

## STEP 1：读取任务和上下文

优先读取：

- `.adpp/changes/{{ feature-dir }}/tasks.md`
- `.adpp/changes/{{ feature-dir }}/design.md`
- `.adpp/changes/{{ feature-dir }}/ux-notes.md`（如果存在且当前任务涉及前端界面）

如果 `ux-notes.md` 存在，必须额外检查：

- 状态是否为 `已确认`
- 是否给出了明确的交互流程
- 是否列出了需要新增/复用的组件
- 是否写明了响应式要求
- 是否填写了设计稿路径（Stitch / Figma / 其他）

读取并展示当前任务状态：
```
📋 当前实现计划：.adpp/changes/{{ feature-dir }}/tasks.md

待完成任务：
  [ ] T001: {{ 描述 }} (预估 {{ time }})
  [ ] T002: {{ 描述 }} (预估 {{ time }})
  ...

可并行执行的任务组：
  组 1: T003 + T005
  组 2: T007 + T008

从哪个任务开始？（输入任务编号，或输入 "all" 按顺序执行）
```

如果当前任务涉及前端界面，额外输出：

```markdown
🎨 前端实现补充输入

- ux-notes: {{ 存在/不存在 }}
- 设计稿路径: {{ 已填写/未填写 }}
- 当前实现基线: {{ tasks.md / ux-notes.md / design file }}
```

### 前端任务的实现优先级

当任务涉及前端界面时，按以下顺序使用设计输入：

1. `tasks.md` 中明确列出的实现任务
2. `ux-notes.md` 中的交互流程、组件清单、响应式要求
3. 可访问的设计稿路径（如 Stitch / Figma）

如果 `ux-notes.md` 与设计稿存在冲突：

- 以最新确认过的 `design.md` / `ux-notes.md` 为准
- 记录冲突点并暂停确认，不得自行假设

### 设计稿路径的使用规则

如果 `ux-notes.md` 中填写了设计稿路径：

- 当前运行环境**能够访问**该链接、并且有权限读取时，可以按设计稿进行实现
- 当前运行环境**不能访问**该链接、链接需要额外登录、或只能看到空白/无权限页面时，不得声称“已按设计稿实现”

此时必须退回以下处理方式之一：

1. 以 `ux-notes.md` 的文字说明为实现基线继续开发
2. 提示用户补充截图、导出稿、标注图或可访问的分享链接
3. 将当前任务标记为“设计输入不足，待补充”

对于你提供的这类 Figma URL：

```text
https://www.figma.com/design/...&node-id=1-3
https://www.figma.com/design/...&node-id=1-315
...
```

只有在当前环境能真正打开这些 node 并读取页面内容时，它们才是可执行输入；否则它们只是“设计引用”，不是可直接实施的设计事实源。

## STEP 2：初始化开发环境

对于每个任务，建议使用 git worktree 隔离：

```bash
# AI 自动执行（需要用户确认）
git worktree add -b feature/{{ task-id }}-{{ slug }} ../{{ project }}-{{ task-id }}
```

执行前输出：
```
即将创建独立开发分支：
  分支名：feature/{{ task-id }}-{{ slug }}
  工作目录：../{{ project }}-{{ task-id }}

确认执行？(yes/no)
```

## STEP 3：TDD 循环（每个任务强制执行）

### 3.1 RED — 先写失败的测试

**规则：在写任何实现代码之前，必须先写测试。**

测试文件命名：
- Python: `tests/test_{{ module }}.py`
- TypeScript: `{{ module }}.test.ts`
- Go: `{{ module }}_test.go`
- Java: `{{ Module }}Test.java`

测试必须覆盖：
- 正常路径（Happy Path）
- 边界条件（Boundary）
- 异常情况（Error Cases）
- 关键业务规则

写完测试后，**必须实际运行测试并确认失败**：
```
▶ 运行测试（应该失败）...
{{ 运行命令 }}

❌ 测试失败（预期行为）：
  {{ 失败原因 }}

确认测试因正确原因失败后，继续写实现代码。
```

**不得声称"测试通过"而不实际运行。**

### 3.2 GREEN — 最小实现

编写**最少的代码**使测试通过。
- 不过度设计
- 不添加未被测试覆盖的功能
- 代码必须符合 `.adpp/spec/coding-standards.md`

写完后运行测试：
```
▶ 运行测试（应该通过）...
{{ 运行命令 }}

✅ 所有测试通过：{{ N }} passed
```

### 3.3 REFACTOR — 重构

在测试绿色的前提下重构：
- 消除重复代码
- 提高可读性
- 优化性能（如有必要）

每次重构后重新运行测试，确保仍然通过。

### 3.4 任务完成检查

```
T{{ id }} 完成检查：
  ✅ 测试已运行并通过
  ✅ 代码符合编码规范
  ✅ 有适当的注释和文档
  ✅ 无明显的安全问题

更新 tasks.md：
  [x] T{{ id }}: {{ 描述 }}（已完成）
```

## STEP 4：危险操作检查

**在执行任何操作之前**，检查是否匹配 `.adpp/config.yaml` 中的 `danger_operations.patterns`：

### 静默模式（mode: silent）
```
⚠️ 检测到危险操作：
  操作：{{ 操作描述 }}
  影响：{{ 影响范围 }}

已记录到：.adpp/workspace/pending-approvals.log

格式：
  时间: {{ timestamp }}
  操作: {{ 命令或 SQL }}
  影响: {{ 描述 }}
  状态: 待审批
```

### 确认模式（mode: confirm，默认）
```
🔴 [危险操作确认] 即将执行：

  {{ 操作的完整命令/SQL }}

  影响范围：{{ 分析影响 }}
  是否可逆：{{ 是/否 }}

请确认：
  "执行" → 继续
  "跳过" → 跳过此操作
  "中止" → 停止整个任务
```

### 移动端推送模式（mode: mobile）
```
📱 危险操作已推送至移动端审批。
   等待移动端确认中...（Webhook: {{ url }}）

推送内容：
  项目: {{ project }}
  操作: {{ 操作描述 }}
  时间: {{ timestamp }}
```

## STEP 5：代码自检

每次提交前，AI 自动检查：

```
🔍 代码自检中...

检查项：
  □ 是否有未处理的 TODO/FIXME
  □ 是否有硬编码的密钥或密码
  □ 是否有注释掉的代码块
  □ 测试覆盖率是否满足规范要求（≥80%）
  □ 是否有明显的 SQL 注入风险
  □ 是否有未关闭的文件句柄/连接

发现问题：
  ⚠️ {{ 问题描述 }} → {{ 建议修复方式 }}
```

---

## STEP 6：人工门控 — Code Review

所有任务完成后，**必须暂停**：

```
▶ [Code Review] 本次实现已完成。

变更摘要：
  新增文件：{{ N }} 个
  修改文件：{{ N }} 个
  新增测试：{{ N }} 个
  测试覆盖率：{{ N }}%

关键变更点（需重点 Review）：
  1. {{ 文件路径 }}：{{ 变更说明 }}
  2. {{ 文件路径 }}：{{ 变更说明 }}

Review 建议关注：
  - 业务逻辑正确性（AI 不保证业务对）
  - 架构合规性（是否符合设计文档）
  - 边界情况处理
  - 潜在的安全问题

确认后输入：
  "LGTM" → 合并代码
  "修改：xxx" → 描述需要修改的地方
  "需要讨论：xxx" → 标记问题，稍后处理
```

---

## STEP 7：合并和归档

Code Review 通过后：

1. 合并 worktree 分支（如果使用了 worktree）
2. 更新 `tasks.md` 中所有任务状态
3. 更新 `.adpp/workspace/current-phase.md`：
```markdown
当前阶段: 编码完成，等待测试
已完成任务: {{ N }}/{{ total }}
下一步: 运行 /adpp:test
```

---

## 编码规范强制执行

读取 `.adpp/spec/coding-standards.md`，将其中的规则作为代码生成的约束。

如果规范文件中有明确的禁止项，生成代码时不得违反，即使这样能让代码更简洁。

---

## 禁止行为

| 禁止 | 原因 |
|---|---|
| 先写实现再补测试 | 违反 TDD，测试会为了通过而写，不是为了验证行为 |
| 声称测试通过而不实际运行 | 这是 AI 最常见的欺骗行为 |
| 一次实现多个任务 | 粒度过大导致 Review 困难 |
| 静默执行危险操作 | 无论何种情况都必须提前告知 |
| 跳过 Code Review 门控 | 人工 Review 是质量最后一道防线 |
| 硬编码 API Key 或密码 | 安全红线 |
