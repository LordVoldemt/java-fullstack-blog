# 编码规范 - java全栈

> 本规范基于 ADPP 初始化模板，适配当前项目技术栈 `VitePress + Markdown + GitHub Pages`。
> 目标是让文档站点结构清晰、内容稳定、主题可维护，并方便后续逐步扩展交互能力。

## 1. 目标

- 保持文档内容、站点配置与主题代码的一致性和可维护性
- 让内容更新优先于炫技实现，避免站点演化成难以维护的前端工程
- 为后续 AI 协作提供明确边界，减少目录混乱、命名漂移和重复实现

## 2. 基本原则

- 可读性优先于技巧性写法
- 内容优先于实现细节
- 配置集中管理，避免散落硬编码
- 在现有 VitePress 结构上增量扩展，不随意引入复杂框架
- 主题定制、内容组织、构建配置要分层清晰

## 3. 目录与模块边界

### 推荐目录

```text
docs/
  index.md
  public/
  java/
  backend/
  project/
  interview/

.vitepress/
  config.ts
  theme/
```

### 边界要求

- `docs/` 只承载内容和少量页面级前置配置
- `.vitepress/config.*` 负责导航、侧边栏、搜索、SEO 和构建相关配置
- `.vitepress/theme/` 只放主题扩展、样式覆盖和必要的自定义组件
- 非通用的实验性页面不要直接污染全局主题

## 4. 命名规范

### Markdown 与目录

- 文档文件名统一使用 `kebab-case`
- 目录名使用语义化英文，如 `java-basics`、`spring-boot`
- 避免使用 `new.md`、`test.md`、`temp.md` 这类无语义名称

### TypeScript / JavaScript

- 变量、函数使用 `camelCase`
- 组件使用 `PascalCase`
- 常量使用 `UPPER_SNAKE_CASE`
- 主题文件名默认使用 `kebab-case`

## 5. 复杂度控制

- 单个主题组件优先控制在 200 行以内
- 单个函数优先控制在 30 行以内
- 参数优先不超过 4 个
- 不为简单需求引入状态管理、复杂组合式封装或额外 UI 框架

## 6. 配置与接口

- 导航、侧边栏、社交链接、SEO 信息集中维护在站点配置中
- 自定义组件的 props 要显式定义，避免隐式传值
- 外部脚本、分析工具、评论系统等第三方集成必须通过配置接入

## 7. 错误处理

- 构建脚本失败时要提供清晰错误信息
- 自定义组件需处理空数据、缺失 props 和资源不存在等情况
- 不允许通过静默吞错掩盖构建或渲染问题

## 8. 日志与可观测性

- 构建阶段至少能从命令行定位错误来源
- 若接入统计或搜索服务，需在文档中注明用途、配置位置和停用方式
- 生产站点不输出敏感配置和调试信息

## 9. 安全基线

- 不在仓库中提交密钥、token、私有 webhook 或后台凭据
- 外链脚本和嵌入内容要说明来源与风险
- 自定义 HTML、脚本注入和评论系统集成需特别审查 XSS 风险

## 10. 内容与代码协作约束

- 文章调整目录结构时，要同步检查导航和侧边栏
- 主题改动前，优先评估是否能通过 Markdown 或默认主题能力解决
- 首页视觉改动不能影响移动端可读性和文档正文浏览体验

## 11. Git 与提交规范

- 采用 Conventional Commits
- 内容更新与主题重构尽量分开提交
- 提交信息要体现影响范围

## 12. AI 协作附加约束

- AI 生成代码前先确认改动是内容型、配置型还是主题型
- AI 不得绕过构建验证直接宣称完成
- AI 不应在没有明确需求时引入大型前端依赖或复杂工程结构
