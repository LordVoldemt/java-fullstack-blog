# 测试规范 - java全栈

> 当前项目以 VitePress 静态博客为主，测试重点不是传统后端单元测试，
> 而是构建可用性、页面结构稳定性、主题改动安全性，以及关键浏览路径的人工验证。

## 1. 测试目标

- 确保文档站点可以稳定构建和发布
- 确保导航、侧边栏、首页和核心文章页可正常访问
- 确保主题扩展不会破坏移动端阅读体验

## 2. 基本原则

- 构建验证是最低门槛
- 内容改动和主题改动都需要可复现的验证方式
- 自动化优先，必要时补人工验证说明
- 不允许仅凭“看起来没问题”宣称完成

## 3. 测试分层

### Build Verification

- 运行 `npm run docs:build`
- 验证静态资源、路由和 Markdown 编译没有报错

### Content Verification

- 检查新增或修改文章的标题层级、代码块、链接和目录展示
- 检查导航与侧边栏是否同步更新

### Theme Verification

- 自定义主题、样式或组件改动后，验证首页和文章页显示正常
- 验证桌面端和移动端的关键展示是否可用

### Manual Smoke Test

- 首页可打开
- 至少一篇文章页可打开
- 导航跳转正常
- 无明显样式错乱

## 4. 质量门槛

- 每次改动至少通过一次 `npm run docs:build`
- 涉及主题或结构改动时，必须补充人工 smoke test
- 涉及链接或目录迁移时，必须检查关键入口没有失效

## 5. 测试命名与记录

- 以功能或变更主题记录验证项，例如“首页重构验证”“Java 分类目录验证”
- 验证结果记录到 `test-report.md` 或 `closeout-note.md`
- 若暂时无法运行验证，必须记录阻塞原因

## 6. 覆盖要求

- 内容型改动：覆盖构建和关键页面检查
- 配置型改动：覆盖构建、导航和信息架构检查
- 主题型改动：覆盖构建、首页、正文页和移动端检查

## 7. 禁止行为

- 不运行构建就宣称通过
- 只检查首页，不检查实际文章页
- 主题改动后不验证移动端
- 链接迁移后不检查入口是否失效

## 8. AI 协作附加约束

- AI 必须真实运行可运行的验证命令
- AI 若受环境限制不能运行验证，要明确写出限制和建议补测项
- AI 不得用弱断言或模糊描述代替验证结果
