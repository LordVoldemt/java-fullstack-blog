# 文档规范 - java全栈

> 本项目是面向 Java 知识分享的 VitePress 博客站点，文档本身就是核心产品。
> 所有内容、结构和说明都应以“便于读者获取知识”和“便于作者长期维护”为优先目标。

## 1. 文档目标

- 让读者能快速理解文章主题、适用场景和实践步骤
- 让内容输出、站点结构和发布流程可追踪、可复用
- 让后续迭代时可以明确区分事实来源文档与派生展示文档

## 2. 文档分层

### Source of Truth

- `.adpp/changes/<feature>/prd.md`
- `.adpp/changes/<feature>/design.md`
- `.adpp/changes/<feature>/ux-notes.md`
- `docs/**` 下的正式文章内容
- `.vitepress/config.*` 中体现的信息架构配置

### Derived Docs

- `README.md`
- 首页介绍文案
- 导航文案、专题入口文案
- 发布说明、阶段性总结

### Evidence Docs

- `closeout-note.md`
- 构建输出目录记录
- 发布验证记录

## 3. 每类核心文档最少结构

### 文章类文档

- 背景或问题
- 核心概念
- 实战步骤或示例
- 注意事项
- 总结或延伸阅读

### PRD

- 背景与目标
- 范围与非范围
- 用户价值
- 验收标准
- 风险与待确认项

### Design

- 信息架构
- 技术方案
- 配置与目录影响
- 风险与回滚思路

## 4. Markdown 与排版规范

- 标题层级不超过四级
- 一篇文章只聚焦一个主题，避免堆砌多条主线
- 代码块必须标注语言类型
- 命令、路径、配置项统一使用代码格式
- 时间统一写为绝对日期，例如 `2026-05-02`
- 长段落优先拆成短段和扁平列表

## 5. 命名规范

- 文章文件名使用 `kebab-case`
- 专题目录使用稳定英文名，避免频繁改名
- 阶段文档统一使用 `prd.md`、`design.md`、`tasks.md`、`ux-notes.md`

## 6. 变更日志与版本管理

- 站点正式发布建议遵循 Semantic Versioning
- 重要改版应记录 Added、Changed、Fixed 等变更摘要
- 涉及目录迁移、URL 变化、SEO 影响时，必须在 closeout 中说明

## 7. 可追溯性要求

- 每轮重要改动至少能追溯到对应的 PRD 或 Design
- 文章结构调整要能追溯到信息架构或导航设计决策
- 主题和配置改动要能追溯到实际用户体验目标

## 8. 首页与站点文案要求

- 首页文案要明确站点定位：Java 学习、实践总结、知识分享
- 导航命名保持稳定，避免同一概念多种叫法
- 专题名称优先用读者能理解的术语，而不是作者内部简称

## 9. closeout 同步要求

- 内容改动完成后，要同步说明更新了哪些文章或结构
- 配置改动完成后，要同步说明影响了哪些导航、侧边栏或构建行为
- 发布前至少记录一次构建验证结果

## 10. 文档质量检查表

- 文档是否回答了“写给谁看、解决什么问题、如何验证有效”
- 文案是否存在术语混乱或重复表达
- 标题和导航是否能支持后续扩展
- 代码示例是否与主题一致且可读
- 是否避免了过时、占位或未标记状态的草稿内容
